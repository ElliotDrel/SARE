## Plan and Feature List

### 1. Project Overview

The application helps individuals collect positive stories from their network, reflect on those stories, and receive a strengths report. Two user groups exist: primary users (people requesting stories) and storytellers (people supplying stories). A lightweight admin role supports moderation and analytics.

### 2. Unified Four-Stage Process Timeline

| Stage | Name | Primary User Actions | System Actions |
|-------|------|---------------------|----------------|
| 1 | Learn and Prepare | Read a short explainer, review tips on who to invite | Display four-stage banner, mark stage complete on click of Next |
| 2 | Collect Stories | Add storyteller contacts, send requests, monitor submissions | Send email and optional SMS, update progress bar, log follow-ups, highlight overdue rows |
| 3 | Self Reflection | Answer guided questions about personal strengths | Auto-save text, allow later editing, unlock Generate Report when saved |
| 4 | Strengths Report | Review compiled report, download PDF, restart cycle if desired | Render report, store PDF, email copy to user |

The banner appears at the top of every authenticated page, always showing the current stage and permitting backward navigation.

### 3. Detailed Feature Set

#### 3.1 User Management
- Email, password, and optional phone registration
- Email verification on sign up
- Password reset link, password strength meter, optional multi factor code via SMS
- Session timeout after 30 minutes of inactivity

#### 3.2 Story Collection Module
- Add Storyteller modal with name, email, phone (optional)
- Prewritten, editable invitation templates for email and SMS
- Single Send Requests button triggers both channels through queued jobs
- Manual resend option shows copyable text but no longer blocks progress
- Default target of ten stories, editable per project (min 5, max 20)
- Table columns: Name, Email, Phone, Status, Last Contact, Submission Date
- Automatic reminders after 3, 7, and 14 days if no submission

#### 3.3 Progress Dashboard
- Circular progress bar and numeric counter
- Red row highlight if no submission after 7 days
- Filter, sort, and search storytellers
- Export dashboard data to CSV

#### 3.4 Storyteller Portal
- One-click account creation using the invitation email as username, password optional
- Save draft capability with autosave every 15 seconds
- Three story fields, first required, others optional
- Word count indicator and sample prompts above each field
- Confirmation page with thank-you note and ability to edit until user locks report

#### 3.5 Self Reflection Module
- Appears only when collected stories ≥ configured target
- Three mandatory text areas: strengths, evidence, growth themes
- Autosave and Save and Continue button
- Cannot exceed 1,500 characters per field (enforced client and server side)

#### 3.6 Report Generation
- Combines self story and all external stories in chronological order
- Browser preview and Download PDF button (A4 and US Letter)
- Embedded charts: word cloud of frequent adjectives, bar chart of strength themes
- Email copy of PDF sent to user and stored in user profile for later access

#### 3.7 Notification Framework
- SendGrid (primary) and Mailgun (fallback) for email
- Twilio for SMS with daily rate limit per account, hard stop at 200 messages
- Message queue (Redis plus Bull) to batch and throttle sends
- Bounce and delivery-failure webhooks update status column and trigger retry logic

#### 3.8 Admin and Analytics
- Admin dashboard lists users, storytellers, stories, and error logs
- Downloadable CSV exports, aggregate charts of completion rates and average time to finish
- Ability to resend invitations or lock abusive accounts
- Role based access control: admin, user, storyteller

#### 3.9 Security, Privacy, and Compliance
- HTTPS everywhere, HTTP disabled
- Argon2 password hashing, 12 character minimum, common-password blacklist
- Role based permissions enforced on every API route
- Consent checkbox on storyteller portal explains data usage
- GDPR, CCPA, and CAN-SPAM compliance statements in footer
- Daily encrypted database backups, seven-day point-in-time recovery

#### 3.10 Accessibility and Internationalization
- WCAG 2.1 AA color contrast and keyboard navigation
- ARIA labels on forms and progress indicators
- Alt text on all icons and images
- Language files ready for English and Spanish, locale auto-detected from browser

### 4. Technical Architecture

#### 4.1 Frontend
- **Framework:** React with TypeScript
- **Build Pipeline:** Vite
- **Routing:** TanStack Router
- **Data Fetching:** React Query
- **Styling:** Tailwind CSS utility classes, Headless UI components
- **Form Validation:** Zod
- **Notifications:** React Hot Toast

#### 4.2 Backend and API
- **Hosting:** Serverless functions on Vercel, written in TypeScript
- **Runtime:** Edge runtime functions for latency-sensitive tasks
- **API:** RESTful routes at /api, versioned through request header
- **Database Access:** Supabase client SDK and REST interface with row-level security
- **Authentication:** JWT session tokens managed by Supabase, stored in secure SameSite cookies
- **Rate Limiting:** Authentication routes limited to one hundred requests per fifteen minutes per IP

#### 4.3 Database Schema (Supabase PostgreSQL)

| Table | Key Columns | Purpose |
|-------|------------|---------|
| users | id, email, password_hash, phone, verified_at | Primary account data |
| storytellers | id, user_id, name, email, phone, token, invited_at | People invited by a user |
| stories | id, storyteller_id, user_id, content, created_at | Submitted stories |
| self_reflections | id, user_id, strengths, evidence, growth, updated_at | User reflections |
| invitations | id, storyteller_id, channel, status, last_sent | Tracks each email or SMS attempt |
| progress | user_id, target_count, collected_count, updated_at | Cached progress data |
| admins | id, email, password_hash, role | Admin accounts |

#### 4.4 Integrations
- **PDF Generation:** Headless Chromium in a Vercel serverless function renders a dedicated report route
- **Charts:** D3 server-side SVG for word clouds and bar charts, embedded in the PDF
- **File Storage:** Supabase Storage buckets store PDF files with signed URLs for download
- **Monitoring:** Vercel observability dashboard, with alerts sent to Slack webhook

#### 4.5 Deployment and Scalability
- **Deployment:** Continuous deployment via Vercel Git integration, automatic preview environments for each pull request
- **Static Assets:** Served globally via Vercel edge network
- **Scalability:** Serverless and edge functions scale automatically based on request volume
- **Database:** Supabase managed Postgres cluster, daily backups and point-in-time recovery, hosted in the United States
- **Containers:** No container orchestration required

### 5. Testing and Quality Assurance

| Layer | Tools | Main Checks |
|-------|-------|-------------|
| Unit | Jest (backend), Vitest (frontend) | Pure functions, service logic |
| Integration | Supertest for API, React Testing Library for components | Endpoint contracts, form interactions |
| End-to-end | Playwright | Full user and storyteller flows on desktop and mobile |
| Accessibility | Axe CLI in CI pipeline | Color contrast, ARIA compliance |
| Load | k6 scripts | 1,000 concurrent users, 2 messages per second sustained |
| User Acceptance | 15 pilot users, 45 storytellers | Onboarding clarity, completion success |

All tests run in GitHub Actions on pull request, blocking merge on failure.

### 6. Data Ownership and Retention
- Stories belong to the requesting user but storytellers can request deletion at any time
- Deleted stories are hard-deleted within 24 hours and removed from backups within 30 days
- Inactive accounts are purged after two years of no logins and no pending projects
